Ch11.14: Win32 File APIs

Overview

On Windows, user-mode applications talk to the kernel through kernel32.dll. The two entry points for opening files are:

  • CreateFileW — the wide-character (UTF-16) variant, used on every modern Windows release (NT, 2000, XP, 7, 10, 11).
  • CreateFileA — the ANSI variant, still encountered on the Windows 9x family (95, 98, ME).

fast_io hides this difference behind two types, mirroring the POSIX layer:

  • ::fast_io::win32_file — an RAII owning wrapper that calls CreateFileW (or CreateFileA on 9x) in its constructor and CloseHandle in its destructor.
  • ::fast_io::win32_io_observer — a non-owning view over an existing HANDLE. Trivially copyable, no destructor.

Choosing Between ANSI and Wide: win32_api_family

The Win32 layer exposes a traits type called ::fast_io::win32_api_family. It selects which variant of the API should be used on a given Windows target:

  • On the NT family (the default for any supported compiler today), the family picks CreateFileW and the wide-character overloads.
  • On the 9x family, the family picks CreateFileA and the ANSI overloads.

In practice you almost never need to touch win32_api_family directly: win32_file consults it automatically when you pass a filename.

Opening a File with win32_file


#include <fast_io.h>
#include <fast_io_device.h>

int main()
{
    using namespace ::fast_io::iomnp;

    // The constructor picks CreateFileW or CreateFileA through
    // ::fast_io::win32_api_family.
    ::fast_io::win32_file wf(u8"hello_win32.txt", ::fast_io::open_mode::out);

    println(wf, u8"Hello from win32_file!");

    ::std::size_t const n{42zu};
    println(wf, u8"The answer is: ", n);
    // CloseHandle is invoked automatically when `wf` goes out of scope.
}

Wrapping an Existing HANDLE

Just like posix_io_observer, you can build a non-owning view over a HANDLE you obtained elsewhere: 


#include <fast_io.h>
#include <fast_io_device.h>

void write_via_handle(void* h)
{
    using namespace ::fast_io::iomnp;

    // Non-owning: does NOT call CloseHandle on destruction.
    ::fast_io::win32_io_observer observer{
        static_cast<::fast_io::win32_io_observer::native_handle_type>(h)};
    println(observer, u8"Written through a win32_io_observer\n");
}

int main()
{
    ::fast_io::win32_file wf(u8"legacy_handle.txt", ::fast_io::open_mode::out);
    write_via_handle(wf.native_handle());
}

Reading and Open-at

Reading uses scan. Opening relative to a directory handle uses NtCreateFile under the hood because the Win32 API itself has no *at equivalent: 


#include <fast_io.h>
#include <fast_io_device.h>

int main()
{
    using namespace ::fast_io::iomnp;

    ::fast_io::win32_file inpf(u8"input.txt", ::fast_io::open_mode::in);
    ::std::size_t value{};
    ::fast_io::io::scan(inpf, value);

    ::fast_io::win32_file outpf(u8"output.txt", ::fast_io::open_mode::out);
    println(outpf, "Read the value: ", value);
}

Windows Also Has File Descriptors

A common misunderstanding is that Windows only has HANDLEs and no file descriptors. This is incorrect. The Windows C runtime (MSVCRT, UCRT) provides POSIX-like file descriptors through functions like _fdopen, _open, _read, _write, and _close. These are declared in <io.h> and <fcntl.h>.

The C runtime maintains a table mapping file descriptors (small integers) to Win32 HANDLEs. You can convert between them:

  • _get_osfhandle(fd) — converts a file descriptor to a Win32 HANDLE. See the Microsoft documentation.
  • _open_osfhandle(handle, flags) — converts a Win32 HANDLE to a file descriptor.
  • _fdopen(fd, mode) — associates a C FILE* stream with a file descriptor. See the Microsoft documentation. 

#include <fast_io.h>
#include <fast_io_device.h>
#include <io.h>
#include <windows.h>

int main()
{
    using namespace ::fast_io::iomnp;

    // Windows C runtime file descriptor (not a HANDLE!)
    int fd{::_open("data.txt", _O_RDONLY)};
    if (fd == -1) {
        // handle error
    }

    // Convert file descriptor to Win32 HANDLE
    void* h{::_get_osfhandle(fd)};

    // Now you can use Win32 APIs on it
    ::fast_io::win32_io_observer observer{
        static_cast<::fast_io::win32_io_observer::native_handle_type>(h)};
    println(observer, "Using Win32 API on a C runtime fd");

    // Close the file descriptor (this also closes the underlying HANDLE)
    ::_close(fd);
}

This proves that Windows has both file descriptors and HANDLEs. The C runtime provides a POSIX-like layer on top of the Win32 API. When you call _read(fd, ...), the C runtime internally calls ReadFile(_get_osfhandle(fd), ...).

The Raw Win32 I/O APIs

Before using fast_io's wrappers, it helps to understand the underlying Win32 APIs. These are the functions exported by kernel32.dll:

API Function Purpose
CreateFileW(lpFileName, dwDesiredAccess, dwShareMode, lpSecurityAttributes, dwCreationDisposition, dwFlagsAndAttributes, hTemplateFile) Open or create a file by name (wide-character / UTF-16). Returns a HANDLE.
CreateFileA(...) ANSI variant of CreateFileW. Used on Windows 9x (95, 98, ME).
ReadFile(hFile, lpBuffer, nNumberOfBytesToRead, lpNumberOfBytesRead, lpOverlapped) Read bytes from a file handle. Returns BOOL; actual bytes read is written to lpNumberOfBytesRead.
WriteFile(hFile, lpBuffer, nNumberOfBytesToWrite, lpNumberOfBytesWritten, lpOverlapped) Write bytes to a file handle. Returns BOOL; actual bytes written is written to lpNumberOfBytesWritten.
CloseHandle(hObject) Close a handle. Always required — handles are a limited resource.
SetFilePointer(hFile, lDistanceToMove, lpDistanceToMoveHigh, dwMoveMethod) Change the file pointer. dwMoveMethod is FILE_BEGIN, FILE_CURRENT, or FILE_END. For large files, use SetFilePointerEx.
GetFileSize(hFile, lpFileSizeHigh) Get the size of a file. For large files, use GetFileSizeEx.

These Win32 APIs are unbuffered — each call goes through kernel32.dll to the kernel. This makes them slow for small or frequent operations. fast_io's buffered types (like ibuf_file and obuf_file) wrap these calls with user-space buffering for much better performance.

Using win32_file for RAII Even Without fast_io I/O

Even if you prefer to call the raw Win32 APIs directly (e.g., ReadFile and WriteFile) instead of using fast_io's print and scan, you should still use ::fast_io::win32_file for RAII. The destructor guarantees CloseHandle is called, so you never forget to close the handle.

This is especially important when exceptions or early returns are involved — without RAII, you must manually call CloseHandle on every exit path, which is error-prone. 


#include <fast_io.h>
#include <fast_io_device.h>
#include <windows.h>

int main()
{
    // Open with win32_file for RAII, but use raw Win32 ReadFile/WriteFile
    ::fast_io::win32_file wf(u8"data.bin", ::fast_io::open_mode::out);

    // Get the raw HANDLE
    void* h{wf.native_handle()};

    // Use raw Win32 WriteFile (unbuffered, goes through kernel32.dll)
    char const buffer[]{"Hello from raw WriteFile\n"};
    DWORD written{};
    BOOL ok{::WriteFile(h, buffer, static_cast<DWORD>(sizeof(buffer) - 1), &written, nullptr)};

    if (!ok || written != sizeof(buffer) - 1) {
        // handle error
    }

    // wf's destructor calls CloseHandle(h) automatically when it goes out of scope.
    // No need to manually call CloseHandle() here, even if we return early or throw.
}

The same principle applies to reading: 


#include <fast_io.h>
#include <fast_io_device.h>
#include <windows.h>

int main()
{
    ::fast_io::win32_file wf(u8"data.bin", ::fast_io::open_mode::in);
    void* h{wf.native_handle()};

    char buffer[256];
    DWORD nread{};
    BOOL ok{::ReadFile(h, buffer, sizeof(buffer), &nread, nullptr)};

    if (ok && nread > 0) {
        // Process the buffer
    }

    // CloseHandle(h) is called automatically
}

For seeking, use SetFilePointerEx on the native handle: 


#include <fast_io.h>
#include <fast_io_device.h>
#include <windows.h>

int main()
{
    ::fast_io::win32_file wf(u8"data.bin",
        ::fast_io::open_mode::in | ::fast_io::open_mode::out);
    void* h{wf.native_handle()};

    LARGE_INTEGER distanceToMove;
    distanceToMove.QuadPart = 100;

    // Seek to offset 100 from the beginning of the file
    BOOL ok{::SetFilePointerEx(h, distanceToMove, nullptr, FILE_BEGIN)};
    if (!ok) {
        // handle error
    }

    // Now read or write from the new position
}

Summary

  • ::fast_io::win32_file wraps the Win32 CreateFileW / CreateFileA entry point with RAII.
  • ::fast_io::win32_io_observer is a non-owning view over an existing HANDLE.
  • ::fast_io::win32_api_family selects between the ANSI and Wide variants based on the target Windows family. You normally do not need to interact with it directly.
  • All standard fast_io algorithms — print, println, scan — work on these types.
  • Windows also has file descriptors through the C runtime (MSVCRT/UCRT), not just Win32 HANDLEs. Use _get_osfhandle(fd) to convert a file descriptor to a HANDLE, and _open_osfhandle(handle, flags) to convert back.
  • The raw Win32 APIs are CreateFileW (or CreateFileA on 9x), ReadFile, WriteFile, CloseHandle, SetFilePointerEx, and GetFileSizeEx. These are unbuffered and go through kernel32.dll.
  • Even if you call the raw Win32 APIs directly (e.g., ReadFile and WriteFile), still use ::fast_io::win32_file for RAII to guarantee CloseHandle is called automatically. This prevents resource leaks, especially with exceptions or early returns.
  • For more details on the Win32 file APIs, see the Microsoft documentation.
