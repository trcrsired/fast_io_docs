Ch11.13: POSIX File APIs

Overview

On every POSIX compliant system (Linux, FreeBSD, macOS, etc.), files are accessed through a small set of system calls: open, openat, read, write, close, and friends. These calls operate on a raw integer file descriptor (int).

fast_io wraps these system calls behind two types:

  • ::fast_io::posix_file — an RAII owning wrapper. It opens the file via open (or openat) and guarantees close is called when the object goes out of scope.
  • ::fast_io::posix_io_observer — a non-owning view over an existing file descriptor. It is trivially copyable, has no destructor, and is safe to pass around by value.

Opening a File with posix_file

The most common case: open a file by name and read/write with print and println. 


#include <fast_io.h>
#include <fast_io_device.h>

int main()
{
    using namespace ::fast_io::iomnp;
    // Open a file for writing. The file is closed automatically
    // when `pf` goes out of scope.
    ::fast_io::posix_file pf(u8"hello_posix.txt", ::fast_io::open_mode::out);

    // fast_io's print/println work directly on posix_file.
    println(pf, u8"Hello from posix_file!");

    ::std::size_t const n{42zu};
    println(pf, u8"The answer is: ", n);
}

posix_file's constructor forwards to open with flags derived from ::fast_io::open_mode. The destructor calls close if the file descriptor is valid.

Wrapping an Existing File Descriptor

Sometimes you receive an int fd from legacy code, a library, or a system call you made directly. You can wrap it without duplicating or taking ownership: 


#include <fast_io.h>
#include <fast_io_device.h>
#include <unistd.h>

int main()
{
    using namespace ::fast_io::iomnp;
    // A raw file descriptor from somewhere else in your program.
    int fd{::open("legacy.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644)};

    // Non-owning observer: does NOT close the fd on destruction.
    ::fast_io::posix_io_observer observer{fd};
    println(observer, u8"Written through a posix_io_observer\n");

    // If you want ownership, construct a posix_file. It will close
    // the fd when it goes out of scope.
    ::fast_io::posix_file owned{::fast_io::from_fd, fd};
    println(owned, u8"Written through an owning posix_file\n");
}

Choose posix_io_observer when the file descriptor's lifetime is managed elsewhere. Choose posix_file when you want RAII cleanup.

Reading with posix_file

Input works through scan, which reads from the underlying file descriptor using read: 


#include <fast_io.h>
#include <fast_io_device.h>

int main()
{
    using namespace ::fast_io::iomnp;
    ::fast_io::posix_file inpf(u8"input.txt", ::fast_io::open_mode::in);

    ::std::size_t value{};
    scan(inpf, value);

    ::fast_io::posix_file outpf(u8"output.txt", ::fast_io::open_mode::out);
    println(outpf, "Read the value: ", value);
}

Open-at Relative to a Directory

Just like obuf_file from earlier chapters, posix_file works with at(dir) to open files relative to a directory file descriptor, using openat under the hood: 


#include <fast_io.h>
#include <fast_io_device.h>

int main(int argc, char** argv)
{
    using namespace ::fast_io::iomnp;
    ::fast_io::posix_file dir{::fast_io::mnp::os_c_str(argv[1]),
                              ::fast_io::open_mode::directory};
    ::fast_io::posix_file pf(at(dir), u8"relative.txt",
                             ::fast_io::open_mode::out);
    println(pf, u8"This file is opened with openat()\n");
}

The Raw POSIX I/O System Calls

Before using fast_io's wrappers, it helps to understand the underlying POSIX system calls. These are the actual kernel entry points (section 2 of the manual):

System Call Purpose
open(path, flags, mode) Open a file by path, returns an int file descriptor.
openat(dirfd, path, flags, mode) Open a file relative to a directory file descriptor. Safer than open because it avoids race conditions from path resolution.
read(fd, buf, count) Read up to count bytes from fd into buf. Returns the number of bytes actually read, or -1 on error.
write(fd, buf, count) Write count bytes from buf to fd. Returns the number of bytes actually written, or -1 on error.
close(fd) Close the file descriptor. Always required — file descriptors are a limited resource.
lseek(fd, offset, whence) Change the file offset. whence is SEEK_SET, SEEK_CUR, or SEEK_END.
fdopen(fd, mode) Associate a C FILE* stream with an existing file descriptor. Returns a FILE*.

These system calls are unbuffered — each call goes directly to the kernel. This makes them slow for small or frequent operations. fast_io's buffered types (like ibuf_file and obuf_file) wrap these calls with user-space buffering for much better performance.

Using posix_file for RAII Even Without fast_io I/O

Even if you prefer to call the raw POSIX APIs directly (e.g., read and write) instead of using fast_io's print and scan, you should still use ::fast_io::posix_file for RAII. The destructor guarantees close is called, so you never forget to close the file descriptor.

This is especially important when exceptions or early returns are involved — without RAII, you must manually call close on every exit path, which is error-prone. 


#include <fast_io.h>
#include <fast_io_device.h>
#include <unistd.h>

int main()
{
    // Open with posix_file for RAII, but use raw POSIX read/write
    ::fast_io::posix_file pf(u8"data.bin", ::fast_io::open_mode::out);

    // Get the raw file descriptor
    int fd{pf.native_handle()};

    // Use raw POSIX write (unbuffered, goes directly to kernel)
    char const buffer[]{"Hello from raw write(2)\n"};
    ::ssize_t written{::write(fd, buffer, sizeof(buffer) - 1)};

    if (written < 0) {
        // handle error
    }

    // pf's destructor calls close(fd) automatically when it goes out of scope.
    // No need to manually call close() here, even if we return early or throw.
}

The same principle applies to reading: 


#include <fast_io.h>
#include <fast_io_device.h>
#include <unistd.h>

int main()
{
    ::fast_io::posix_file pf(u8"data.bin", ::fast_io::open_mode::in);
    int fd{pf.native_handle()};

    char buffer[256];
    ::ssize_t nread{::read(fd, buffer, sizeof(buffer))};

    if (nread > 0) {
        // Process the buffer
    }

    // close(fd) is called automatically
}

For seeking, use lseek on the native handle: 


#include <fast_io.h>
#include <fast_io_device.h>
#include <unistd.h>

int main()
{
    ::fast_io::posix_file pf(u8"data.bin",
        ::fast_io::open_mode::in | ::fast_io::open_mode::out);
    int fd{pf.native_handle()};

    // Seek to offset 100 from the beginning of the file
    ::off_t result{::lseek(fd, 100, SEEK_SET)};
    if (result == static_cast<::off_t>(-1)) {
        // handle error
    }

    // Now read or write from the new position
}

Summary

  • ::fast_io::posix_file is an RAII wrapper around POSIX open / close.
  • ::fast_io::posix_io_observer is a lightweight, non-owning view over an existing file descriptor.
  • Both types plug directly into ::fast_io::io::print, ::fast_io::io::println, and ::fast_io::io::scan.
  • You can wrap any raw int fd without changing the rest of your code.
  • The raw POSIX system calls are open, openat, read, write, close, lseek, and fdopen. These are unbuffered and go directly to the kernel.
  • Even if you call the raw POSIX APIs directly (e.g., read and write), still use ::fast_io::posix_file for RAII to guarantee close is called automatically. This prevents resource leaks, especially with exceptions or early returns.
  • For more details on POSIX APIs, see the Linux man pages or your system's documentation.
