Ch11.15: NT File APIs

Overview

Modern Windows (NT, 2000, XP, 7, 10, 11) is built on the NT kernel. The lowest-level user-mode entry points for file I/O are the native system calls:

  • NtCreateFile
  • NtReadFile
  • NtWriteFile
  • NtClose

These live in ntdll.dll and are the same functions the Win32 subsystem itself calls. A parallel set of entry points with the Zw prefix (ZwCreateFile, …) exist as well; they differ only in whether the previous-mode parameter is set to KernelMode or UserMode. For user-mode code they behave identically.

fast_io exposes the NT layer through:

  • ::fast_io::nt_file — an RAII owning wrapper. The constructor invokes NtCreateFile; the destructor calls NtClose.
  • ::fast_io::nt_io_observer — a non-owning view over an existing NT HANDLE. Trivially copyable, no destructor.

The programming model is exactly the same pattern as posix_file and win32_file.

Opening a File with nt_file


#include <fast_io.h>
#include <fast_io_device.h>

int main()
{
    using namespace ::fast_io::iomnp;

    // The constructor translates open_mode into the corresponding
    // NtCreateFile parameters (DesiredAccess, CreateDisposition, ...).
    ::fast_io::nt_file nf(u8"hello_nt.txt", ::fast_io::open_mode::out);

    print(nf, "Hello from nt_file!\n");

    ::std::size_t const n{42zu};
    println(nf, u8"The answer is: ", n);
    // NtClose is invoked automatically when `nf` goes out of scope.
}

Wrapping an Existing HANDLE


#include <fast_io.h>
#include <fast_io_device.h>

void inspect(void* nt_handle)
{
    using namespace ::fast_io::iomnp;

    // Non-owning view. No NtClose on destruction.
    ::fast_io::nt_io_observer observer{
        static_cast<::fast_io::nt_io_observer::native_handle_type>(nt_handle)};

    println(observer, u8"Written through an nt_io_observer\n");
    println(::fast_io::mnp::handlevw(observer.native_handle()));
}

int main()
{
    ::fast_io::nt_file nf(u8"inspect.txt", ::fast_io::open_mode::out);
    inspect(nf.native_handle());
}

Relative Opens

Because NtCreateFile natively accepts a RootDirectory handle and a relative ObjectName, the NT layer is the most natural home for at(dir) on Windows: 


#include <fast_io.h>
#include <fast_io_device.h>

int main(int argc, char** argv)
{
    using namespace ::fast_io::iomnp;

    ::fast_io::nt_file dir{::fast_io::mnp::os_c_str(argv[1]),
                           ::fast_io::open_mode::directory};
    ::fast_io::nt_file nf(at(dir), u8"relative.txt",
                          ::fast_io::open_mode::out);
    println(nf, u8"Opened through NtCreateFile with a root directory\n");
}

Why Use nt_file Directly?

Most programs should just use ::fast_io::native_file (which aliases to nt_file on NT) or ::fast_io::iobuf_file. You reach for nt_file directly when you need:

  • Direct kernel access without going through kernel32.dll.
  • Fine-grained control over NtCreateFile parameters.
  • Relative opens, which map cleanly onto the native API.

The Raw NT I/O APIs

Before using fast_io's wrappers, it helps to understand the underlying NT native APIs. These are the functions exported by ntdll.dll, the lowest-level user-mode entry points that kernel32.dll itself calls:

API Function Purpose
NtCreateFile(FileHandle, DesiredAccess, ObjectAttributes, IoStatusBlock, AllocationSize, FileAttributes, ShareAccess, CreateDisposition, CreateOptions, EaBuffer, EaLength) Open or create a file. Returns a HANDLE through FileHandle. This is the most fundamental file-opening API on NT.
NtReadFile(FileHandle, Event, ApcRoutine, ApcContext, IoStatusBlock, Buffer, Length, ByteOffset, Key) Read bytes from a file handle. Can be synchronous or asynchronous.
NtWriteFile(FileHandle, Event, ApcRoutine, ApcContext, IoStatusBlock, Buffer, Length, ByteOffset, Key) Write bytes to a file handle. Can be synchronous or asynchronous.
NtClose(Handle) Close a handle. Always required — handles are a limited resource.
NtSetInformationFile(FileHandle, IoStatusBlock, FileInformation, Length, FileInformationClass) Set file information, including file position (seek). Used with FilePositionInformation.
NtQueryInformationFile(FileHandle, IoStatusBlock, FileInformation, Length, FileInformationClass) Query file information, including file position and size.

There are also Zw* variants (ZwCreateFile, ZwReadFile, etc.). In user space, Nt* and Zw* are identical. They differ only in kernel mode: Nt* validates parameters from user mode, while Zw* sets the previous-mode to KernelMode and skips validation. For user-mode code, they behave the same.

These NT APIs are unbuffered — each call goes through ntdll.dll to the kernel (ntoskrnl.exe). This makes them slow for small or frequent operations. fast_io's buffered types (like ibuf_file and obuf_file) wrap these calls with user-space buffering for much better performance.

Using nt_file for RAII Even Without fast_io I/O

Even if you prefer to call the raw NT APIs directly (e.g., NtReadFile and NtWriteFile) instead of using fast_io's print and scan, you should still use ::fast_io::nt_file for RAII. The destructor guarantees NtClose is called, so you never forget to close the handle.

This is especially important when exceptions or early returns are involved — without RAII, you must manually call NtClose on every exit path, which is error-prone. 


#include <fast_io.h>
#include <fast_io_device.h>
#include <ntstatus.h>

// Note: NtCreateFile, NtReadFile, etc. are declared in the Windows SDK headers.
// fast_io provides wrappers, but you can also call them directly.

int main()
{
    // Open with nt_file for RAII, but use raw NT NtReadFile/NtWriteFile
    ::fast_io::nt_file nf(u8"data.bin", ::fast_io::open_mode::out);

    // Get the raw HANDLE
    void* h{nf.native_handle()};

    // In practice, calling NtWriteFile directly is complex due to IO_STATUS_BLOCK
    // and OBJECT_ATTRIBUTES setup. Most programs should use fast_io's print/println.
    // This example shows the concept, but real code typically uses fast_io's I/O.

    // nf's destructor calls NtClose(h) automatically when it goes out of scope.
    // No need to manually call NtClose() here, even if we return early or throw.
}

Note: Calling NT APIs directly is complex. NtCreateFile requires setting up OBJECT_ATTRIBUTES, IO_STATUS_BLOCK, and many other parameters. NtReadFile and NtWriteFile are similarly involved. Most programs should use fast_io's print, println, and scan instead. The raw APIs are documented here for completeness and to show that fast_io wraps them with RAII.

The same RAII principle applies if you do call the raw NT APIs: 


#include <fast_io.h>
#include <fast_io_device.h>

int main()
{
    ::fast_io::nt_file nf(u8"data.bin", ::fast_io::open_mode::in);
    void* h{nf.native_handle()};

    // If you call NtReadFile directly here...
    // (complex setup omitted for brevity)

    // NtClose(h) is called automatically by nf's destructor
}

Summary

  • ::fast_io::nt_file wraps the NT kernel's NtCreateFile / NtClose with RAII.
  • ::fast_io::nt_io_observer is a non-owning view over an existing NT HANDLE.
  • The API surface mirrors posix_file and win32_file: once you know one layer, the others are trivial.
  • On modern Windows, ::fast_io::native_file is an alias for nt_file.
  • The raw NT APIs are NtCreateFile, NtReadFile, NtWriteFile, NtClose, NtSetInformationFile, and NtQueryInformationFile. These are unbuffered and go through ntdll.dll to the kernel.
  • Nt* and Zw* variants are identical in user space. They differ only in kernel mode.
  • Even if you call the raw NT APIs directly (e.g., NtReadFile and NtWriteFile), still use ::fast_io::nt_file for RAII to guarantee NtClose is called automatically. This prevents resource leaks, especially with exceptions or early returns.
  • Calling NT APIs directly is complex. Most programs should use fast_io's print, println, and scan instead.
  • For more details on the NT APIs, see the Microsoft WDM documentation.
